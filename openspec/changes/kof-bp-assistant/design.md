## Context

全新 Android 应用，无既有代码库。目标平台 Android 8.0+，优先适配小米 MIUI。应用在 BP 阶段（游戏大厅，非对局内）激活，进入对局后完全停止录屏，规避 TP 反外挂检测。

核心约束：
- 后台内存占用 < 50MB
- 截图→识别→展示结果 < 500ms
- 进入对局后 MediaProjection 必须停止
- 不读取游戏进程/内存，不模拟操作

## Goals / Non-Goals

**Goals:**
- BP 阶段手动截图，识别对方禁/选英雄（pHash）
- 输出轻量级阵容/套路意图提示、阵容级反制思路和对位克制建议
- 悬浮窗可折叠、可拖动，进入对局后隐藏
- 启动时版本检测，强制更新数据包

**Non-Goals:**
- 自动检测 BP 阶段（不用无障碍服务持续监听）
- 识别己方英雄或提供己方建议（MVP 不做）
- 复杂机器学习/大模型阵容推理（MVP 仅做本地规则匹配）
- iOS 支持
- Google Play 上架

## Decisions

### D1：识别算法选 pHash，不引入 OpenCV

**选择**：自实现 pHash（感知哈希，64-bit），不依赖 OpenCV for Android。

**理由**：OpenCV AAR ~20MB，而 pHash 核心逻辑 < 200 行 Kotlin。BP 界面英雄头像是固定图标，无复杂特征提取需求，pHash 汉明距离匹配精度足够。

**替代方案**：OpenCV 模板匹配（精度略高，但包体大）；TFLite 分类模型（精度最高，~50MB，杀鸡用牛刀）。

---

### D2：英雄哈希库由开发者预建，打包进 App

**选择**：开发者在自己设备上截取所有英雄头像，提取 pHash，生成 `hero_hashes.json`，打包进 App 或随数据包下发。

**理由**：不让用户做任何校准操作。pHash 对图像缩放有鲁棒性，统一 resize 到 32×32 后哈希值在不同设备间差异极小（实测汉明距离 < 5）。

**版权处理**：APK 内只存 `{hero_id: pHash_int64}` 键值对，不包含任何图片资源文件。

---

### D3：截图用 MediaProjection，手动触发，BP 结束后停止

**选择**：用户点"扫描"按钮触发一次截图；用户点"进入游戏"后调用 `mediaProjection.stop()`。

**理由**：事件驱动零轮询，后台几乎零 CPU 占用。进入对局后停止录屏是规避 TP 检测的硬性要求。

**替代方案**：无障碍服务监听（王者荣耀用 Unity 渲染，无障碍服务抓不到游戏内部节点，不可行）。

---

### D4：坐标配置热更新，按屏幕比例存储

**选择**：英雄槽位坐标存在 `layout_config.json`，用屏幕宽高百分比表示，随数据包从服务端下发。

**理由**：王者荣耀 UI 随版本可能变化，热更新坐标无需重新发版 App。比例坐标自动适配同纵横比的不同分辨率。

---

### D5：后台服务用 Foreground Service + 电池白名单引导

**选择**：启动一个带通知栏图标的 Foreground Service 维持悬浮窗存活；首次启动引导用户将 App 加入 MIUI 电池白名单（省电策略 → 无限制）。

**理由**：MIUI 会激进地杀掉后台普通 Service，Foreground Service 是 Android 官方机制中存活率最高的方案。

---

### D6：数据更新采用全量替换

**选择**：每次更新下载完整数据包（zip），解压覆盖本地文件。

**理由**：英雄数量约 120，克制关系 JSON 总大小 < 200KB，全量简单可靠，无需维护增量 diff 逻辑。

---

### D7：阵容意图识别采用轻量规则引擎

**选择**：MVP 使用本地 JSON 规则匹配，不引入 TFLite、大模型或服务端实时推理。规则由 `combos.json`、`season_meta.json`、`counter_strategies.json` 组成：
- `combos.json`：定义常见阵容/套路标签、触发英雄集合、ban/pick 权重、最少命中数和置信度阈值
- `season_meta.json`：定义当前赛季强势英雄、强势阵容、英雄强度梯度和版本说明
- `counter_strategies.json`：定义命中某类阵容后的反制思路、推荐英雄标签和避免选择的英雄类型

**理由**：BP 阶段需要快速响应和低内存驻留。规则匹配可以在几十到几百条规则内完成，数据常驻内存后耗时可控，且可以通过数据包热更新跟随赛季变化。

**匹配方式**：每次扫描后，对已识别的敌方 ban/pick 英雄计算规则分数：
- pick 命中核心英雄加高权重
- ban 命中保护性禁用或针对性禁用加中权重
- 当前赛季强势英雄/阵容加版本权重
- 命中数不足或分数低于阈值时输出"意图不明显"

**性能约束**：规则数量 MVP 控制在 200 条以内，分析阶段目标耗时 < 50ms，内存增量 < 5MB。

## Risks / Trade-offs

- **pHash 跨版本失效** → 游戏 UI 改版导致头像渲染变化，pHash 大量失配。缓解：数据包包含 pHash 库，版本更新时同步更新哈希值，强制更新机制保障用户及时更新。

- **MIUI 版本碎片化** → MIUI 14 / HyperOS 对悬浮窗和后台策略行为不一致。缓解：针对常见小米机型（小米13/14 系列、Redmi Note 系列）测试兼容性；权限引导页覆盖各版本路径。

- **MediaProjection 授权弹窗体验差** → Android 每次创建 VirtualDisplay 前需弹系统授权弹窗（部分设备每次都弹）。缓解：引导用户"记住选择"；Session 内只请求一次授权，复用同一个 MediaProjection 实例直到用户点"进入游戏"。

- **坐标偏移导致识别失败** → 不同 MIUI 版本状态栏高度不同，ROI 计算偏移。缓解：坐标配置中加入状态栏高度补偿字段；提供"识别校准"入口让用户手动微调。

- **数据建议质量参差** → 克制关系数据由开发者手动维护，可能存在主观偏差或版本滞后。缓解：每个赛季更新一次数据包；界面注明数据版本，用户知晓时效性。

- **规则误判阵容意图** → 对方可能临时补位或故意迷惑 ban/pick，轻量规则无法保证高置信推理。缓解：UI 展示"可能意图"和置信度等级，仅在命中核心英雄组合时给强提示；低置信时回退到单英雄克制建议。

## Migration Plan

全新应用，无迁移需求。发布流程：

1. 打包 APK → 上传到自有文件服务器
2. 更新服务端 `version.json`（`force_update: false` 软更新，或 `true` 强制）
3. 用户下次启动自动检测并提示更新

回滚：服务端 `version.json` 回退到上一版本号，旧 APK 继续可用。

## Open Questions

- **pHash 实际精度**：不同小米机型截图的英雄头像渲染是否有足够一致性？需要在 2～3 款主流小米设备上实测汉明距离分布后确认阈值（推荐初始阈值：汉明距离 ≤ 10）。
- **MediaProjection 授权时机**：是 App 启动时提前申请，还是用户点"扫描"时再申请？前者体验流畅但感觉"提前索权"，后者更合理但首次有延迟。
- **规则数据来源**：当前赛季强势英雄和阵容数据由开发者人工维护，还是接入第三方统计源后半自动生成？MVP 可先人工维护，后续再评估自动化。
